OpenAI выложила в открытый доступ модель Privacy Filter — детектор персональных данных в тексте. Лицензия Apache 2.0, веса доступны на Hugging Face, для коммерческого использования ограничений нет.

Модель решает задачу token-level классификации: для каждого токена предсказывает, относится ли он к PII, и к какому классу. Поддерживаются имена, телефоны, email-адреса, физические адреса, номера документов, банковские реквизиты, IP-адреса, а также секреты вроде API-ключей и приватных токенов. Всего 27 классов.

Архитектурно это MoE-модель на 1.5B параметров с 50M активных. Такая конфигурация даёт быстрый инференс на CPU и компактных GPU. На Apple M3 модель обрабатывает около 8000 токенов в секунду, на одной RTX 4090 — порядка 60 тысяч.

По метрикам OpenAI приводит F1 97.43% на собственном бенчмарке из 50 тысяч размеченных примеров. На публичном датасете AI4Privacy результат скромнее — F1 94.1%, но всё равно выше специализированных предшественников на 3–5 пунктов. Особенно заметен прирост на смешанных языках: русский, арабский, хинди модель держит почти так же уверенно, как английский.

В репозитории сразу лежат три варианта:

- Полные веса в bf16 для дообучения.
- Квантизация Q4 для CPU-инференса (около 900 МБ).
- ONNX-сборка для интеграции в браузерные пайплайны.

Использовать модель можно через стандартный `transformers` или через готовую утилиту `privacy-filter` на PyPI. Утилита принимает текст, возвращает либо размеченный JSON со спанами, либо готовую маскированную строку с заменами вида `[NAME_1]`, `[PHONE_2]`. Замены детерминированы: одна и та же сущность в одном документе получает один и тот же индекс.

Контекст применения очевиден: предобработка датасетов для обучения, фильтрация логов, чистка пользовательских обращений перед отправкой во внешние LLM. OpenAI отдельно отмечает кейс с RAG-системами — модель можно поставить между retrieval и генерацией, чтобы из выдачи не утекали PII из сторонних документов.

Из ограничений стоит держать в голове два. Первое: модель плохо работает на коде — там слишком много ложных срабатываний на переменные с именами вроде `user_name`. Для кода OpenAI рекомендует отдельный набор регулярок. Второе: контекст ограничен 8192 токенами; для длинных документов нужен скользящий буфер с перекрытием.

Дообучение под собственные классы PII описано в README. Достаточно нескольких сотен размеченных примеров, чтобы добавить новый тип сущности — например, внутренние идентификаторы клиентов или коды продуктов.
